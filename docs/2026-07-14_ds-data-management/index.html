Concept 1

なぜデータ基盤を三層に分けるのか

データ基盤は一般に「データソース → データレイク → データウェアハウス（DWH） → データマート」という段階を経て整備されます。それぞれの層には明確な役割があります。

層 役割 メリット
データレイク データを「そのままコピー」して1カ所に集める 格納先確認などのコストが下がり、組織横断的な分析が可能になる
データウェアハウス（DWH） 「共通化されたデータ」を置く 各所での後続処理を省け、誰でも同じ定義の指標（例: 売上）を参照できる
データマート 加工処理した「特定の目的のデータ」を置く すぐに集計結果を確認でき、応答時間が速く、他用途への影響を気にせず自由に更新できる

具体例

データレイクで「そのまま」コピーするのは、分析テーマによって適切な加工方法が異なるためです。最初から特定の目的に加工してしまうと、レイクの段階で情報が失われ、後から別の分析テーマに使い回せなくなります。

判断の軸: 「このデータをどれだけ多くの用途に使い回したいか」が、レイク（汎用）・DWH（共通化）・マート（特定目的）のどこに置くべきかを決める軸になる。

理解チェック

「売上」の定義が部署によってバラバラだったチームが、DWH層を整備したとします。これによって解消される問題を、マート層だけを整備した場合と比較して説明してください。

Concept 2

メタデータとは何か：DAMA-DMBOK2とAikenのピラミッド

データマネジメントの標準的な定義として、DAMA-DMBOK2は次のように述べています。

DAMA-DMBOK2の定義

「データとインフォメーションという資産の価値を提供し、管理し、守り、高めるために、それらのライフサイクルを通して計画、方針、スケジュール、手順などを開発、実施、監督することである」

この定義は範囲が広いため、資料では「Aikenのピラミッド」を使って位置づけを整理しています。土台にデータマネジメントがあり、その上にデータ分析、さらにその上にビッグデータ活用が積み上がるという階層構造です。つまり、派手な分析・活用の前提として、地味な整備（データマネジメント）が土台になっているという考え方です。

メタデータ（「データを説明するためのデータ」）は、一般に3種類に分類されます。

分類 内容の例
ビジネスメタデータ そのデータが何を意味するか、誰が使うか、業務上の定義
テクニカルメタデータ カラムの名称・型、テーブルの構造、変換ロジック
オペレーショナルメタデータ 抽出結果の履歴、実行ログ、アクセス権限、更新頻度
判断の軸: メタデータ整備で困ったときは「これはビジネス・テクニカル・オペレーショナルのどれを説明する情報か」を切り分けると、抜け漏れに気づきやすい。

理解チェック

「このカラムの値は税込みか税抜きか分からない」という問題は、3分類のうちどのメタデータが不足している状態でしょうか。

Concept 3

実践①：BigQueryでカラムの説明を管理する

もっとも手軽なメタデータ管理は、データマートを作成するコード自体に説明を埋め込むことです。BigQueryでは、テーブル作成時にカラムのdescriptionを直接指定できます。

CREATE TABLE `project.dataset.sales_mart` (
  order_id STRING OPTIONS(description="注文ID。ソースシステムの注文番号と1:1対応"),
  sales_amount_jpy NUMERIC OPTIONS(description="税込み売上金額（円）。返品時はマイナス値"),
  order_date DATE OPTIONS(description="注文日。JST基準")
)
OPTIONS(description="日次の店舗別売上を集計したデータマート");

具体例

「税込みか税抜きか」「タイムゾーンはJSTかUTCか」といった、資料で例示されていた曖昧になりがちな情報を、カラムの説明としてスキーマ自体に埋め込んでおくと、クエリを書く人がBigQueryのコンソール上で確認でき、口頭確認や過去のドキュメントを探す手間が減ります。

判断の軸: ドキュメントを別ファイル（Wiki等）で管理すると更新が追いつきにくい。コードとメタデータを同じ場所（同じPR）で更新できる仕組みにしておくと、鮮度が保ちやすい。

Concept 4

実践②：dbt-osmosisによるメタデータの自動伝播

データマートが増えてくると、同じカラム（例: order_id）の説明を、上流モデルから下流モデルへ何度も手で書き写す必要が出てきます。dbt-osmosisはこの手間を解消するツールで、公式ドキュメントには次のように説明されています。

公式ドキュメントの記載

「カラムドキュメントの継承機能は、リネージ（データの系譜）に基づいて、上流モデルのカラムの説明・タグ・その他メタデータを下流モデルへ自動的に伝播させる」

代表的なコマンドは次の3つです。

# スキーマYAMLファイルの構成を整理する
dbt-osmosis yaml organize

# 上流モデルの説明を下流モデルへ伝播させる
dbt-osmosis yaml document

# organize/documentなどをまとめて実行する（推奨コマンド）
dbt-osmosis yaml refactor

具体例：CI/CDへの組み込み

公式ドキュメントでは、dbt-osmosis yaml refactor -Cをpre-commitフックとして組み込む例が紹介されています。これにより、開発者が新しいモデルを追加してコミットするたびに、既存の説明が自動的に下流へ伝播し、「上流の説明だけ更新して下流を直し忘れる」というメタデータの陳腐化を防げます。

# .pre-commit-config.yaml の例（イメージ）
- repo: local
  hooks:
    - id: dbt-osmosis-refactor
      name: dbt-osmosis yaml refactor
      entry: dbt-osmosis yaml refactor -C
      language: system
      pass_filenames: false
判断の軸: 「同じカラムが複数モデルに登場するか」「説明を書く場所が上流と下流でズレやすいか」が、dbt-osmosis導入の効果を判断する軸になる。手動でのコピペ運用が3箇所を超えたあたりから導入価値が出やすい。

理解チェック

upstream（ステージング層）のモデルでカラムの説明を1つ更新し、dbt-osmosis yaml documentを実行したとします。このカラムを参照している下流のマートが3つあった場合、手作業のときと比べて何が変わるか説明してください。

Concept 5

スモールスタートという考え方

三層構造やメタデータ管理は理想形として整理されていますが、資料では「最初から本格的に作らなくてよい」ことも強調されています。「ドライブのディレクトリ分け＋元データコピー」から始めることも容認される、という主張です。

具体例

BigQueryやdbt環境を用意する前でも、共有ドライブに「レイク相当（生データのコピー置き場）」「マート相当（集計済みの共有フォルダ）」のようにディレクトリを分けるだけで、三層構造の考え方の一部を先取りできます。ツール導入は後からでも構いません。

判断の軸: 「今すぐ完璧な基盤を作れるか」ではなく「今の混乱を一段階でも整理できるか」を基準に、身の丈に合った一歩から始める。

Concept 6

木こりのジレンマ：なぜDSが推進役になるべきか

データ整備を提案しても、「データを整備するともっと早く分析できますよ」に対して「分析が忙しくてそんな暇はないんだよ」という反応が返ってくることがあります。これが「木こりのジレンマ」（斧を研ぐ時間がなく、切れない斧で木を切り続ける寓話になぞらえたもの）です。

解決の方向性

資料が提示する解決策は、いきなり全社的な整備計画を立てるのではなく、まず試験運用の段階で具体的な成果物（例: 説明付きのデータマート）を使ってもらい、価値を実感してもらうことです。「整備の議論」より先に「整備された小さな成果物」を見せる、というアプローチです。

そのうえで資料は、DS自身がデータマネジメントの推進役になるべき理由を2つ挙げています。

理由 内容
ケイパビリティ データに一番詳しいのはDS自身であり、何が整備されていないかを具体的に指摘できる
インセンティブ データが綺麗になって一番最初に恩恵を受けるのはDS自身（分析が速く・正確になる）
判断の軸: 「誰が整備すべきか」の議論で止まっている場合、「一番困っていて一番詳しい人（多くはDS自身）から小さく始める」という順番に切り替えると前進しやすい。

理解チェック

あなたのチームで「データ整備を提案しても後回しにされる」状況があるとしたら、木こりのジレンマのどの部分（忙しさ／価値の未実感／推進役の不在）が一番の障害になっていると考えられますか。

三層構造・メタデータ管理の使い分け早見表

状況 とるべきアクション
データがまだ各所に散らばっている まずはデータレイク相当（1カ所への「そのまま」集約）から着手する
同じ指標（売上など）の定義が部署ごとに違う DWH相当（共通化されたデータ）の整備を優先する
特定の用途向けにすぐ使えるデータが欲しい データマート層で目的別に加工する
同じカラムの説明を複数箇所に手でコピペしている dbt-osmosisなどによる説明の自動伝播を検討する
整備の提案が「忙しいから後で」と後回しにされる 試験運用で小さな成果物を見せ、DS自身が推進役になる

適用判断の手順

  1. 今のデータがどこに散らばっていて、どこまで「そのまま」集約できているかを確認する。
  2. 同じ指標の定義がチーム・部署間で食い違っていないかを洗い出す（DWH整備の必要性の確認）。
  3. 頻繁に使うデータマートについて、まずBigQueryのカラム説明（description）から埋めていく。
  4. 同じカラムの説明を3箇所以上手でコピペしている場合は、dbt-osmosisの導入を検討する。
  5. 整備の提案が後回しにされる場合は、全体計画より先に、小さく整備した成果物を試験運用で見せる。
  6. 整備がチームに定着したら、次の対象（別のマート、別のメタデータ分類）へ広げる。

まとめ・次のステップ

データ基盤を「データレイク（そのままコピー）→DWH（共通化）→データマート（目的別加工）」の三層に分ける考え方は、それぞれの層に異なる役割を持たせることで、データの使い回しやすさと、特定用途への最適化を両立させるための設計思想です。メタデータ管理は、この基盤を長期的に使えるものにするための「データを説明する情報」の整備であり、BigQueryのカラム説明のような手軽な方法から、dbt-osmosisによる自動伝播のような仕組み化まで、段階的に強化できます。そして、こうした整備は「木こりのジレンマ」に陥りがちなため、データに一番詳しく、整備の恩恵を一番早く受けるDS自身が、小さな成果物から推進役になることが有効です。

次のステップとしては、自分が普段使っているデータマートのカラムのうち、説明が曖昧なものを1つ選んでBigQueryのdescriptionを埋めてみること、そして複数モデルで説明のコピペが発生している箇所があればdbt-osmosisのyaml documentコマンドを試してみることをおすすめします。
