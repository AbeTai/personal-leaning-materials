検索最適化サービスの判断軸

クラスタリングキーだけでは拾いにくい検索パターンへの使い方を整理します。

前提知識

  • Snowflakeでテーブルを作成し、`SELECT ... WHERE` を実行できる
  • 等値検索、範囲検索、カーディナリティという用語を聞いたことがある
  • Query Profileで実行時間や読み込み量を確認する目的が分かる

学習ロードマップ

  1. Phase 1 マイクロパーティションとプルーニングを押さえる

    Snowflakeが不要なパーティションを読み飛ばせる条件を理解します。

  2. Phase 2 クラスタリングキーで局所性を作る

    検索条件に合うようにデータ配置を寄せる設計を学びます。

  3. Phase 3 検索最適化サービスで点検索を補う

    検索アクセスパスを使うべき検索パターンを見極めます。

  4. Phase 4 適用判断を言語化する

    性能、コスト、メンテナンスの観点で使い分けを決めます。

Concept 1

マイクロパーティションは「読み飛ばせるか」が重要

Snowflakeはテーブルをマイクロパーティションに分け、それぞれに列値の範囲などのメタデータを持たせます。クエリ条件に合わないと判断できるパーティションは読まれず、これがプルーニングです。

具体例

日付順に投入されたデータでは、日付条件は読み飛ばしやすい一方、顧客IDや価格のように投入順と関係が薄い列では、多くのパーティションを読む可能性があります。

判断の軸: その検索条件で、Snowflakeが「読まなくてよい範囲」をメタデータから判断できるか。

理解チェック

同じテーブルでも、日付検索は速く、顧客ID検索は遅いことがあります。どちらの列が投入順やデータ配置と相性がよいかを確認してください。

Concept 2

クラスタリングキーは、よく使う検索条件にデータ配置を寄せる

クラスタリングキーを指定すると、Snowflakeの自動クラスタリングがバックグラウンドでマイクロパーティションの配置を整えます。よく使う`WHERE`条件とキーが合っているほど、プルーニングが効きやすくなります。

alter table orders
  cluster by (date_trunc(month, o_orderdate), round(o_totalprice, -4));

具体例

`o_orderdate`と`o_totalprice`でよく絞り込むなら、その列をそのまま使うのではなく、月単位や価格帯に丸めて粒度を調整します。粒度が粗すぎると絞れず、細かすぎると配置が安定しにくくなります。

判断の軸: 何度も使われる範囲検索や複合条件があり、粒度を調整できる列・式か。

理解チェック

クラスタリングキーに高カーディナリティなIDをそのまま入れる前に、値をまとめる式や別の検索条件で代替できないか考えてください。

Concept 3

検索最適化サービスは、少数ヒットの探索に強い

検索最適化サービスは、通常のマイクロパーティションメタデータとは別に検索アクセスパスを構築します。クラスタリングキーとは異なる列、全文検索、半構造化データ、地理空間検索などに使えます。

alter table orders
  add search optimization on equality(o_custkey);

具体例

顧客IDの等値検索のように、条件に合う行が少ない検索では、検索アクセスパスによって読むパーティションを大きく減らせます。一方、ヒット件数が多い条件では効果が薄くなります。

判断の軸: 条件に合う行が少なく、クラスタリングキーに入れにくい列や式か。

理解チェック

建物名やIDで一点検索する画面と、広い価格帯で一覧する画面では、どちらが検索最適化サービスに向いているか説明してください。

使い分け早見表

観点 クラスタリングキー 検索最適化サービス
向く検索 頻出する範囲検索、複合条件 少数ヒットの等値検索、全文検索、半構造化検索
効き方 データ配置を整えてプルーニングしやすくする 検索アクセスパスで対象パーティションを見つける
注意点 キーの粒度設計が必要 Enterprise Edition以上と追加ストレージを考慮する
確認方法 `SYSTEM$CLUSTERING_INFORMATION` と Query Profile `SHOW TABLES` の進捗・容量と Query Profile

適用判断の手順

  1. 遅いクエリの`WHERE`条件とQuery Profileを確認する。
  2. 読み込んでいるパーティション数が多いかを見る。
  3. 頻出する範囲検索ならクラスタリングキーの粒度を設計する。
  4. 少数ヒットの探索なら検索最適化サービスを検討する。
  5. 改善後もコスト、ストレージ、メンテナンスの増加を確認する。

まとめ

巨大テーブルの検索性能は、どれだけ不要なマイクロパーティションを読まずに済むかで大きく変わります。クラスタリングキーは頻出検索に合わせてデータ配置を整える手段で、検索最適化サービスはクラスタリングだけでは対応しにくい少数ヒットの探索を補う手段です。

最初にクエリパターンを観察し、次にパーティション読み込み量を確認し、最後にコストを含めて適用する。この順序で考えると、機能名からではなく、検索の性質からチューニングを選べます。
