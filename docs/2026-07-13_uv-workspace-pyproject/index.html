uv initしたら親のpyproject.tomlが勝手に書き換わった話

モノレポのサブディレクトリでuv initを実行すると、なぜ親のpyproject.tomlがワークスペースとして書き換わるのか。検出の仕組み・実務での回避策・モノレポ設計への応用まで踏み込んで整理する教材です。

対象
uvの基本コマンド（uv init/add/run）は使ったことがあるPython開発者

この教材で学べること

何が起きたか

モノレポでbackendを独立化しようとした際に、なぜ親の設定が書き換わったのかを再現します。

ワークスペース検出の仕組み

uvが親ディレクトリのどこまで、何を根拠に探索するのかを公式ドキュメントで確認します。

実務での回避策と設計判断

--no-workspaceの使い方、既存メンバーの外し方、モノレポ設計そのものの判断基準まで扱います。

前提知識の確認

  • uv init / uv add / uv run など、uvの基本コマンドを使ったことがある
  • pyproject.tomlがPythonプロジェクトの設定ファイルであることを知っている
  • モノレポ（1つのリポジトリに複数のサービス・パッケージを置く構成）という言葉の意味が分かる

  2. Phase 2 uvのワークスペース自動検出の仕組みを理解する

    公式ドキュメントの記載をもとに、uvが何を根拠に親ディレクトリを検出しているかを確認します。

  3. Phase 3 実務での回避策・対処法を身につける

    ワークスペースを使うべきケースと、独立プロジェクト（Docker個別化）にすべきケースを整理します。

Concept 1

何が起きたか：ディレクトリ構造で再現する

モノレポのルートにpyproject.tomlがあり、その配下のbackend/ディレクトリで独立したPythonプロジェクトを作ろうとしたケースです。

monorepo/
├── pyproject.toml        ← 既存のルート設定（この時点ではワークスペースではない）
└── dev/
    └── backend/           ← ここで uv init を実行したい

dev/backend/でuv initを実行すると、uvはルートのpyproject.tomlを検出し、以下のように書き換えます。

# 実行前のルート pyproject.toml には無かったものが追加される
[tool.uv.workspace]
members = ["dev/backend"]

具体例

著者が意図していたのは「backendを独立したDockerイメージとしてビルドし、他のサービスと依存関係を共有しない」構成でした。しかし実際には、ルートのpyproject.tomlが勝手にワークスペースの親（ルート）として扱われ、backendがその一員として登録されてしまいました。

判断の軸: サブディレクトリでのuv initは「そのディレクトリだけの初期化」ではなく、「親も含めたプロジェクト全体構造への変更」になり得ることを前提に扱う必要がある。

理解チェック

ルートにpyproject.tomlが存在しない状態でdev/backend/でuv initした場合も、同じ問題は起きるでしょうか。「親ディレクトリの探索」が何を見つけたときに発動するのかを考えて説明してください。

Concept 2

ワークスペース自動検出の仕組み

uv公式ドキュメント（docs.astral.sh）には、この挙動について次のように明記されています。

公式ドキュメントの記載

"By default, running uv init inside an existing package will add the newly created member to the workspace, creating a tool.uv.workspace table in the workspace root if it doesn't already exist."

さらにプロジェクト探索全般について: "All pyproject.toml, uv.toml, and .python-version files will be discovered by walking up the directory tree from the project root, as will the project's virtual environment (.venv)."

つまりuvは、現在のディレクトリから親方向へディレクトリツリーを遡って、pyproject.toml・uv.toml・.python-version・.venvを探索します。親にpyproject.tomlが見つかると、そこが「ワークスペースのルート」とみなされ、今回作成したプロジェクトが自動的にそのメンバーとして登録されます。

判断の軸: 「親ディレクトリにpyproject.tomlが存在するかどうか」が、ワークスペース自動登録が発動するかどうかの分岐点。存在しなければ、通常は独立プロジェクトとして初期化される。

理解チェック

もしモノレポのルートにpyproject.tomlではなくuv.tomlだけが置かれていた場合、同じ探索ロジックによってワークスペース化が起きると考えられますか。上の引用のどの部分から判断できるか答えてください。

Concept 3

実務での回避策・対処法

回避策は大きく2パターンあります。「これから作るとき」と「すでにワークスペースに組み込まれてしまったとき」です。

# パターン1: これから作る場合は --no-workspace を付ける
cd dev/backend
uv init --no-workspace

公式ドキュメントでの--no-workspaceの説明: "Avoid discovering a workspace and create a standalone project. By default, uv searches for workspaces in the current directory or any parent directory."

# パターン2: すでにメンバーとして登録されてしまった場合
# ルートの pyproject.toml から該当メンバーを exclude するか、members から取り除く

[tool.uv.workspace]
members = ["packages/*"]
exclude = ["dev/backend"]   # 個別に除外する場合

具体例

すでにmembersにdev/backendが追加され、ルートのuv.lockにbackendの依存関係が混ざってしまっている場合は、excludeで除外したあとに、dev/backendディレクトリで改めてuv init --no-workspace（または手動で独立したpyproject.tomlを用意）し、uv lockをそのディレクトリ単体で実行し直す必要があります。

判断の軸: 「作る前に防ぐ」なら--no-workspace、「すでに混ざってしまった状態を分離する」ならexclude設定とロックファイルの作り直しの2段階が必要。

理解チェック

excludeだけを設定して、dev/backendディレクトリに独自のuv.lockを作り直さなかった場合、何が起きると考えられますか。「ロックファイルがどこにあるか」という観点で説明してください。

Concept 4

ロックファイル・仮想環境はどこまで共有されるか

ワークスペースの本質的な特徴は「設定ファイルは個別、実行環境は共有」という点です。公式ドキュメントには次のようにあります。

公式ドキュメントの記載

"in a workspace, each package defines its own pyproject.toml, but the workspace shares a single lockfile, ensuring that the workspace operates with a consistent set of dependencies."

つまり、各メンバーは自分の依存関係をpyproject.tomlに書けますが、実際にインストールされるバージョンを固定するuv.lockはワークスペース全体で1つだけです。同様に、仮想環境（.venv）もワークスペースルートに1つだけ作られ、全メンバーで共有されます（ディレクトリツリーの探索で.venvも遡って発見される、という前述の記載と対応します）。

リソース ワークスペース 独立プロジェクト（--no-workspace）
pyproject.toml メンバーごとに個別 個別
uv.lock ワークスペース全体で1つ プロジェクトごとに個別
.venv ワークスペースルートに1つ、共有 プロジェクトごとに個別
判断の軸: 「同じバージョンの依存関係で統一したいか」「Dockerイメージを完全に分離したいか」で、ワークスペースの共有特性が有利にも不利にもなる。

理解チェック

backendとfrontendサービスが同じワークスペースに属していて、backendだけが特定ライブラリの新バージョンを必要とする場合、共有ロックファイルはどのような制約を生みますか。

Concept 5

モノレポ設計への応用：ワークスペースを使うべきか

元記事の結論は「共有コード・統一依存管理をしたいならワークスペース、独立Dockerデプロイをしたいなら独立プロジェクト」というものでした。これをチェックリスト化すると実務判断がしやすくなります。

観点 ワークスペースが向く 独立プロジェクトが向く
パッケージ間の依存 共通ライブラリを複数サービスが参照する サービス間で依存関係を共有しない
デプロイ単位 まとめてデプロイ、または依存関係を揃えたい サービスごとに個別のDockerイメージ
CI/CD 統一的なテスト・ビルドパイプライン サービスごとに最適化された軽量ビルド
依存バージョン管理 全メンバーで統一したい サービスごとに独立して更新したい
判断の軸: 「複数サービス間でコード・依存を共有する必要があるか」を最初に問い、必要ないならuv init --no-workspaceを既定にしておくと、今回のような意図しない書き換えを防げる。

Concept 6

実務チェックリスト：新しいディレクトリでuvを使う前に

  1. 今からPythonプロジェクトを作るディレクトリの親方向に、pyproject.toml・uv.tomlが存在しないか確認する（lsやfind .. -maxdepth 3 -name pyproject.tomlなど）。
  2. 親に設定ファイルがあり、かつ依存関係やロックファイルを共有したくない場合は、必ずuv init --no-workspaceを使う。
  3. 実行後、親のpyproject.tomlに差分が出ていないかgit diffで確認する。
  4. Dockerで個別コンテナ化する予定のサービスは、独立プロジェクトとして.venv・uv.lockが専用に作られているかを確認する。
  5. すでに意図せずワークスペースに組み込まれていた場合は、exclude設定とロックファイルの作り直しで分離する。

具体例：差分確認コマンド

cd dev/backend
uv init --no-workspace
cd ../..
git status
git diff pyproject.toml   # ルートのpyproject.tomlが変化していないことを確認

使い分け早見表

観点 デフォルト（ワークスペース自動検出） --no-workspace
ルートpyproject.tomlへの影響 [tool.uv.workspace]にメンバー追加され書き換わる 変更なし
ロックファイル ワークスペース全体で統合管理（1つ） プロジェクトごとに個別作成
仮想環境 ワークスペースルートで共有 独立
向いているケース 共有コード・統一依存管理をしたいモノレポ 独立してDocker化したいマイクロサービス構成

適用判断の手順

  1. 新しいPythonサービス・パッケージを追加するディレクトリの親方向に既存のpyproject.tomlがないか確認する。
  2. 他サービスとの依存共有・統一デプロイが必要かを整理する。
  3. 必要なければuv init --no-workspaceで初期化し、git diffでルート設定への影響がないことを確認する。
  4. 必要ならワークスペースのままにし、共有ロックファイル・共有.venvが意図通りかを確認する。
  5. 誤ってワークスペースに組み込まれていた場合はexclude設定と個別ロックファイルの作り直しで分離する。

まとめ・次のステップ

uvはサブディレクトリでuv initを実行した際、親方向にディレクトリツリーを遡ってpyproject.tomlなどを探索し、見つかった場合はデフォルトでそこをワークスペースルートとして扱い、新規プロジェクトをメンバーとして自動登録します。この挙動自体は「複数パッケージで依存関係を統一管理したい」場合には便利ですが、独立したDockerサービスとして分離したい場合には意図しない書き換えとして表面化します。--no-workspaceフラグはこの自動検出を明示的に無効化するもので、実務では「親にpyproject.tomlがあるかどうか」を初期化前に確認する習慣と組み合わせることで、今回のような問題を未然に防げます。

次のステップとしては、実際に自分の手元でモノレポ構造を再現し、ワークスペースあり・なしそれぞれでuv initを実行してgit diffと.venv・uv.lockの場所を見比べてみることをおすすめします。余裕があれば、exclude設定を使って既存メンバーを分離する手順も試してみてください。
