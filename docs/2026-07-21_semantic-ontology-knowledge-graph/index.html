AIエージェント時代のコンテキスト整備

前提知識の確認

  • dbtやLookerなどでセマンティックレイヤー機能（指標・ディメンションの定義）を使ったことがある
  • データレイク/DWH/データマートの三層構造、メタデータ（ビジネス/テクニカル/オペレーショナル）の分類が分かる
  • リレーショナルデータベースでテーブル同士をJOINして集計するクエリを書いたことがある

この教材は、セマンティックレイヤーは使用経験があるものの、オントロジーやグラフDBといった隣接概念は初めてという方を想定しています。既知の概念（三層構造・メタデータ）の説明は前回の教材を前提に簡潔にし、新しい概念、特にグラフDB・ナレッジグラフに重点を置いています。

Concept 1

なぜ今「セマンティック」が重要なのか

元資料では、AIエージェントが業務の暗黙知（コンテキスト）を持たないまま数値だけを見て誤った解釈をしてしまう例が示されています。遊園地の売上データ分析を題材にした3つの誤りです。

AIエージェントの解釈 実際の理由
休日なのに売上が0 実は休園日だった
平日なのに売上が急増 実は開園10周年イベントだった
「昨年度売上◯億円」（WHERE year_month BETWEEN '2025-04' AND '2026-03'） 実は決算期が6月で、期間の切り方がずれていた
判断の軸: AIエージェントに数値やクエリを渡すだけでは、業務の「意味」（休園日、記念イベント、決算期）は伝わらない。この「意味」を構造化して渡す仕組みが、セマンティック・オントロジー・ナレッジグラフといった技術群の共通の狙いになる。

理解チェック

「税込み/税抜きの定義が不明確」という、前回の教材で扱ったメタデータの問題と、今回の「決算期のズレ」の問題は、どちらも「AIエージェントに渡すべき暗黙知」という点で共通しています。この2つの問題を解決するには、それぞれどのようなメタデータ・コンテキストを整備すればよいか考えてください。

Concept 2

統制語彙の階層：シソーラス・オントロジー・タクソノミー・フォークソノミー

似ている用語を整理すると、大きく「トップダウンの統制語彙」と「ボトムアップのフォークソノミー」に分かれます。

用語 定義 アプローチ
シソーラス 同じ意味、似た意味、包含関係、上位・下位をまとめたもの トップダウン（統制語彙）
同義語リスト 似た意味の言葉をまとめたもの トップダウン（統制語彙）
オントロジー（従来型） 語彙の関係性をツリー状にまとめたもの トップダウン（統制語彙）
タクソノミー 階層構造を持った分類（大カテゴリ＞中カテゴリ＞小カテゴリなど） トップダウン（統制語彙）
フォークソノミー 利用者が必要に応じてラベルを自由に付ける ボトムアップ

具体例

Wikipediaのタグ付けはフォークソノミーの代表例です。実務では、あらかじめ決めた「公式ラベル」（統制語彙）と、現場が自由に付ける「通常ラベル」（フォークソノミー）を併用し、多くの人が同じラベルを使うようになったら公式ラベルへ昇格させる、という運用が可能です。

判断の軸: 最初から完璧な統制語彙を作ろうとすると整備が止まりやすい。フォークソノミーで自由にラベル付けを始め、使われ方を見てから統制語彙に昇格させる順序が現実的。

理解チェック

「オントロジー」と「タクソノミー」はどちらも階層的ですが、表の定義の違いから、両者の決定的な違いを1文で説明してください。

Concept 3

セマンティックレイヤーのおさらいと、オントロジーの進化

すでに使ったことのあるセマンティックレイヤー（dbt/Looker/Snowflakeなど）は、「ファクトとディメンションをどう組み合わせてどのような集計をするのかを設定ファイルで定義し、参照するための機能」です。「商品の売上とはこの数値を指す」という定義がされることで、「これが公式の売上だ」と宣言しているのと実質的に同じ効果を持ちます。

従来との違い

2000年頃のBIツールでも、ストアドプロシージャのようにSQLをカプセル化する役割は果たされていました。違いは、専用ツールとしてファクト・ディメンションの組み合わせルールを一元管理できること、そしてAIエージェントにコンテキストとして渡しやすい形で定義が構造化されている点です。

一方「オントロジー」は、従来は語彙の関係性を管理する複雑な体系（シソーラスに近いもの）として扱われてきましたが、最近ではPalantir Foundryなどが推進する形で、エンタープライズ・アーキテクチャ（EA）に近い意味合いに変化しています。つまり、単なる用語集ではなく「業務・システム・データを包括的に整理する設計図」としてのオントロジーです。

判断の軸: セマンティックレイヤーは「指標の意味」を統一する仕組み、最近型オントロジーは「業務・システム・データ全体の関係」を統一する仕組みという射程の違いで捉えると、両者の関係が整理しやすい。

Concept 4

グラフDBとは何か：ノード・エッジとリレーショナルDBとの違い

グラフの基本構造は、ノード（◯）とエッジ（→）による表現です。例えば「パスタ」というノードと「人物」というノードを、「作った」「美味しい」といったエッジ（関係性）で結びます。

リレーショナルDBとの違い

リレーショナルデータベースで「ノードA→ノードB→ノードC」のような多重参照（友達の友達、取引先の取引先、など）をたどるには、JOINを繰り返す回数不定形のループ処理が必要になりがちです。グラフDBでは「ノードA・エッジ・ノードB」という3列を持つ1レコードとして関係を表現するため、多段の関係をたどる処理に強い設計になっています。

-- グラフの最小構成イメージ（GQL系クエリ言語で一般的なパターンマッチのイメージ。厳密な構文は製品ごとに異なる）
MATCH (person)-[:MADE]->(dish:Pasta)
RETURN person.name, dish.name

従来はNeo4jやAmazon Neptuneのような専用のグラフDBが必要でしたが、Google Cloud Next '26では「BigQuery Graph」が発表され、既存のBigQueryテーブル（ノードテーブル・エッジテーブル）からETLやデータ複製なしにグラフ分析ができるようになりました。ISO GQL標準・ISO Property Graph Queries (SQL/PGQ)標準に対応したクエリインターフェースを備えています。

判断の軸: 「何段階も関係をたどる必要があるか」「関係性そのもの（エッジ）に意味やスコアを持たせたいか」がリレーショナルDBとグラフDBを使い分ける軸になる。

理解チェック

「Aさんの友達の友達の友達」のようなクエリを、通常のリレーショナルDBのJOINで書こうとすると何が起きますか。グラフDBのMATCH的な表現と比較して説明してください。

Concept 4-2

大規模データを扱うための隣接行列

ノードの数が多くなると、ノードとエッジのリストを逐一たどる処理は計算量が大きくなります。そこで大規模データ処理では、グラフを「隣接行列」という行列形式に置き換えて扱うアプローチが近年の主流になっています。

具体例：3人の関係を隣接行列にする

Aさん・Bさん・Cさんの間の「知り合い」関係（Aさん-Bさん、Bさん-Cさんが知り合い）を隣接行列で表すと、次のようになります。

Aさん Bさん Cさん
Aさん 0 1 0
Bさん 1 0 1
Cさん 0 1 0

行列の掛け算（積）を使うと、「2ステップ先まで知り合いを辿った関係」を一括で計算できます。ノードの数が数百万になっても、行列演算はGPU・専用ライブラリでの高速化と相性がよく、逐次的にエッジをたどるよりも大規模データに向いています。

判断の軸: ノード数・エッジ数が少ないうちはノード・エッジのリストのままで十分だが、大規模になるほど隣接行列（や、それに準じた行列演算ベースの表現）への変換を検討する。

Concept 5

ナレッジグラフの活用事例：消費者-飲食店の支払いグラフ

「意味やナレッジを正確に表現しようとするとグラフ構造になる」というのが、ナレッジグラフの基本的な考え方です。例えば「AさんはBさんちの犬が好き」という関係を誤解なく正確に表現するには、単純な表よりもグラフ構造が向いています（「Aさん」「Bさん」「犬」というノードと、「好き」「飼い主」というエッジの組み合わせで表現できる）。

具体例：消費者-飲食店の支払いグラフ

元資料で紹介されている研究事例では、消費者ノードと飲食店ノードを「支払い」エッジで結んだグラフが構築されています。このグラフを時系列で見ると、COVID-19パンデミック前後でエッジ（支払いの頻度・金額）が細くなる様子が可視化でき、飲食店利用の減少を定量的に捉えることができます。

この種のナレッジグラフ・支払いグラフは、以下のような分析に応用できます。

  • 与信分析: 取引ネットワーク上の位置関係から信用度を推定する
  • 市場予測: 消費行動グラフの変化から需要の変化を先読みする
  • コミュニティ分割によるレコメンド: グラフ上で「近い」消費者・店舗をクラスタ化し、似た嗜好の商品を推薦する
判断の軸: 「関係性の変化そのもの」（誰が誰と繋がっているか、その繋がりが強まっているか弱まっているか）が分析対象になる場合、単純な集計表よりナレッジグラフ表現の方が現象を捉えやすい。

理解チェック

自分の業務データの中に「ノードとエッジ」で表現すると分かりやすくなりそうな関係性（例: 顧客と担当者、部品と製品、部署間の連携など）があるか考えてみてください。

Concept 6

AIエージェント・RAGとセマンティックの関係

元資料では、AIエージェントに関わる用語を次のように整理しています。

用語 説明
AI推論モデル WEB文章等を学習したもの。プロンプトを与えられると推論結果を返す
プロンプト 明示的な単発の指示文
コンテキスト 暗黙的に読まれる一連の情報（プリセットされたルール・ファイルなど）
ハーネス 「承認ボタンを押さないとメールソフトへアクセスできない」など、システムの振る舞いを制御する仕組み
コンテキストレイヤー 社内ドキュメント・体制図・会議体・マニュアル等の非構造化データを統合し、コンテキストとして自動的に読み込ませる仕組み全体

RAG（Retrieval-Augmented Generation）は、生成AIモデルがWEBコンテンツをもとに一般的なオントロジ（知識体系）をすでに内部に持っていることを前提に、「1から国語辞典を作る」ステップを省略し、そこに自社特有の情報を検索して上乗せする手法です。「生成AIの持つ一般的な知識」＋「自社特有の情報」の組み合わせが現在の主流のアプローチです。

判断の軸: セマンティックレイヤー（指標の意味の統一）、統制語彙・オントロジー（用語・業務知識の整理）、ナレッジグラフ（関係性の整理）は、いずれも「AIエージェントに正しいコンテキストを渡す」という同じ目的のための異なる手段として位置づけられる。

理解チェック

Concept 1の「決算期のズレ」の例をコンテキストレイヤーで解決するとしたら、どのような情報を、どのタイミングでAIエージェントに読み込ませればよいか考えてください。

技術要素の使い分け早見表

技術要素 整理する対象 主な用途
データカタログ・用語集 データ資産一覧、用語の定義 「どこに何があるか」を探せるようにする
セマンティックレイヤー（dbt/Looker） ファクトとディメンションの組み合わせ・指標定義 「売上」などの指標の意味を統一する
統制語彙（シソーラス/タクソノミー等） 語彙・分類の階層 用語や分類の表記ゆれ・同義語を整理する
オントロジー（最近型・Palantir型） 業務・システム・データ全体の関係 組織横断でエンタープライズ・アーキテクチャを整理する
グラフDB・ナレッジグラフ 個体間の多段の関係性 与信・市場予測・レコメンドなど関係性そのものを分析する

適用判断の手順

  1. 今困っているのが「指標の定義がバラバラ」なら、まずセマンティックレイヤーの整備・拡充を優先する。
  2. 「同じ意味の言葉が複数の表記で使われている」なら、統制語彙（シソーラス・タクソノミー）とフォークソノミーの併用を検討する。
  3. 「業務・システム・データの全体像が誰も把握できていない」なら、最近型オントロジー（EA的なアプローチ）の整理を検討する。
  4. 「多段階の関係（友達の友達、取引先の取引先など）を分析したい」なら、グラフDB・ナレッジグラフを検討する。
  5. AIエージェントに業務判断をさせたい場合は、以上の整備状況を「コンテキストレイヤー」としてどう読み込ませるかを設計する。

まとめ・次のステップ

データレイク・DWH・マートという三層構造とメタデータ管理は「データそのもの」を整理する仕組みでした。今回学んだセマンティックレイヤー・統制語彙・オントロジー・グラフDB・ナレッジグラフは、そこからさらに一歩進んで「データの意味」「用語の関係」「個体間の関係性」を整理し、人間だけでなくAIエージェントにも正しく伝わる形にするための技術群です。特にグラフDB・ナレッジグラフは、リレーショナルDBが苦手とする多段の関係性分析に強く、与信・市場予測・レコメンドといった実務応用に直結します。

次のステップとしては、自分の業務データの中で「ノードとエッジ」として表現すると分かりやすくなりそうな関係性を1つ具体的に書き出してみること、そして既存のセマンティックレイヤー定義を見直し、AIエージェントに読み込ませる「コンテキストレイヤー」としてどう整備できそうかを考えてみることをおすすめします。
