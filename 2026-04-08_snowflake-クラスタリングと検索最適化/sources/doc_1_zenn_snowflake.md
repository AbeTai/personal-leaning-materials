## はじめに

### 高速化が必要になった背景

estie では 10 億を超えるレコードを持つテーブルを含む不動産データ加工パイプラインを Snowflake 上に構築しています。不動産データの特徴として、データの流通経路が様々で誤ったデータが混入することが多々あります。具体的には似た文字の打ち間違い、桁間違い、単位間違いなど理由も様々です。

主要な物件や統計的におかしいと検知されたレコードなどは Snowflake に直接接続されたデータダッシュボード上で人手で確認・修正を行っています。

これまでは最大でも数百万、数千万レコード程度のデータしか扱っていなかったため特にチューニングせずとも応答速度に問題はありませんでした。しかし、レコード数が 10 億を超えるデータも扱うようになるとデータ読み込みに数十秒待つことが発生しました。そこで、テーブルに対するチューニングが必要になりました。

### Snowflake のテーブル構造・マイクロパーティション

Snowflake では一般的な RDBMS のキーによるクラスタリングとも従来のデータウェアハウスのパーティションとも異なる、**マイクロパーティション**（Micro-partitions）というデータ構造を用いています。

個々のマイクロパーティションは含まれる行の各列の値の範囲などといったメタデータを持っており、データを読む際にメタデータの値だけを見て不要であればテーブルデータ本体の読み込みをスキップする**プルーニング**が行われます。テーブルデータ本体はそこまで高速でないストレージに保存されているため、クエリの高速化においてプルーニングは大変重要です。

Snowflake のマイクロパーティションは自動で分割が行われるため、RDBMS で必要なプライマリキーやデータウェアハウスのパーティション定義のような**事前の定義が必要ない**という特徴があります。

各マイクロパーティションへはテーブルへデータが挿入された順序により分割されます。このテーブルに対してクエリを行うと、メタデータを用いてプルーニングが行われます。

| クエリ | 読み込むマイクロパーティション | プルーニング |
|--------|--------------------------------|------------|
| `where date = '11/5'` | 4 | ⭕ |
| `where name = 'Y'` | 3, 4 | 🔺 |
| `where type = 2` | 1, 2, 3, 4 | ❌ |

自動でマイクロパーティションが分割された場合のプルーニングはデータを挿入した際の順序に依存しています。

### サンプルデータでの検証

Snowflake のサンプルデータセットにある TPC-H のうち最大のデータセットを使いマイクロパーティションの挙動を検証しました。TPCH_SF1000 は 15 億レコードを含んでおり、estie で扱うデータと似たようなスケールを持っています。

```sql
create table orders as select * from snowflake_sample_data.tpch_sf1000.orders;

select * from orders where o_orderdate = '1996-04-23';
select * from orders where o_totalprice between 10000 and 10100;
select * from orders where o_custkey = 133047718;
```

| クエリ | Partitions scanned | 行数 | 実行時間 | プルーニング |
|--------|-------------------|------|---------|------------|
| `where o_orderdate = '1996-04-23'` | 743 / 2,429 | 623.5k | 2.4 秒 | ⭕ |
| `where o_totalprice between 10000 and 10100` | 2,429 / 2,429 | 337k | 43 秒 | ❌ |
| `where o_custkey = 133047718` | 2,429 / 2,429 | 21 | 6.6 秒 | ❌ |

## テーブルのクラスタリング

事前の定義がない場合には分割が挿入順序に依存しており、クエリとの組み合わせによってはプルーニングできないことがありました。

そのような場合には明示的に**クラスタリングキー**（Clustering Keys）を指定することで、マイクロパーティションの分割方法を変更することができます。

クラスタリングキーはテーブルやマテリアライズドビューに対して指定することができ、単一の列、複数の列の組み合わせ、列の値に一部の関数を適用した値などを使うことができます。

適切にクラスタリングされたテーブルに対してのクエリでは**クラスタリングキーでの絞り込みでプルーニングできる可能性が上がります**。

| クエリ | 読み込むマイクロパーティション | プルーニング |
|--------|--------------------------------|------------|
| `where date = '11/5'` | 8 | ⭕ |
| `where type = 2` | 5, 8 | ⭕ |

クラスタリングキーの粒度に関わらずマイクロパーティションの大きさは変わらないという点に注意が必要です。

以上から、次のことが言えます：

- クラスタリングキーのカーディナリティが低すぎる（粗い）と、プルーニングをしきれずパフォーマンスが低下する
- クラスタリングキーのカーディナリティが高すぎる（細かい）と、マイクロパーティションを一定の状態にできずパフォーマンスが低下する

すなわち、クラスタリングキーには適切な粒度が存在し、高すぎても低すぎてもパフォーマンスには悪影響があります。

### サンプルデータでの検証

```sql
alter table orders cluster by (date_trunc(month, o_orderdate), round(o_totalprice, -4));

select system$clustering_information('orders');
```

クラスタリングキーを設定しても即座にクラスタリングが変更されるわけではなく、自動クラスタリングサービスによりバックグラウンドで変更されていきます。

**SYSTEM$CLUSTERING_INFORMATION の結果：**

```json
{
  "cluster_by_keys" : "LINEAR(date_trunc(month, o_orderdate), round(o_totalprice, -4))",
  "total_partition_count" : 2543,
  "total_constant_partition_count" : 40,
  "average_overlaps" : 9.7169,
  "average_depth" : 6.3795
}
```

クラスタリング後の検証結果:

| クエリ | Partitions scanned | 行数 | 実行時間 | プルーニング |
|--------|-------------------|------|---------|------------|
| `where o_orderdate = '1996-04-23'` | 34 / 2,543 | 623.5k | 1.9 秒 | ⭕ |
| `where o_totalprice between 10000 and 10100` | 386 / 2,543 | 337k | 4.1 秒 | ⭕ |

`o_totalprice` では **43秒 → 4.1秒** と10倍の高速化。

## 検索最適化サービス

**検索最適化サービス**（Search optimization service）は RDBMS におけるインデックスに類似しており、特定の型の列や式でのフィルタリングを高速化します。

> 検索最適化サービスの利用には Enterprise Edition 以上が必要です

クラスタリングキーによる高速化ではマイクロパーティションごとのメタデータを用いてプルーニングを行いましたが、検索最適化サービスでは**検索アクセスパス**と呼ばれるデータを生成し利用することでプルーニングを行います。

以下のようなケースで検索最適化サービスによる高速化が期待できます：

- すでにクラスタリングキーが設定されているが、それとは異なる列や式での検索を行いたいとき
- 条件に合致する行が含まれるマイクロパーティションが少なく、かつマイクロパーティション内のデータの分散が大きいなど通常のメタデータだけではプルーニングできないとき
- 複雑な式でフィルタリングを行うとき（全文検索・半構造化データの検索・地理空間検索など）

### サンプルデータでの検証

```sql
alter table orders add search optimization on equality(o_custkey);

show tables like 'orders';
```

最適化の方法：
- `full_text` - 全文検索
- `equality` - 等値比較
- `substring` - 部分文字列・正規表現
- `geo` - 地理空間検索

検索最適化後の検証結果:

| クエリ | Partitions scanned | 行数 | 実行時間 | プルーニング |
|--------|-------------------|------|---------|------------|
| `where o_custkey = 133047718` | 21 / 2,543 | 21 | 0.8 秒 | ⭕ |

**6.6秒 → 0.8秒** と約10倍の高速化。

## 結論

クラスタリングキーを設定するとマイクロパーティションの分割方法を変えることができ、検索対象となるデータの局所性を高めマイクロパーティションのメタデータを用いたプルーニングを行うことでクエリの高速化が可能です。

検索最適化サービスはマイクロパーティションのメタデータとは別に検索アクセスパスを構築する仕組みで、検索対象となるデータが含まれるマイクロパーティションが十分に少ないときにプルーニングを行うことができクエリの高速化が可能です。

これらの最適化を実際に使った結果、estie のデータダッシュボードは 10 億を超えるレコードを持ったテーブルに対しても数秒で検索や修正ができるようになりました。
